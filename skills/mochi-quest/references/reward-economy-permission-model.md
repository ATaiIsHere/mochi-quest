# Reward economy: permission model

Mochi Quest coins are not real money and should not be framed as buying goods. Coins buy **permission / indulgence / constraint relief**.

## Core distinction

- Bad framing: “Spend coins to buy coffee / tea / an app / a meal.”
- Good framing: “Spend coins to unlock permission to do something the user would normally constrain, delay, or feel guilty about.”

If a reward still costs real money, the user pays that money separately. The coin reward only grants a self-management permission.

## What should not be a reward

Do not list things the user can already freely do without special permission, especially neutral low-friction actions such as:

- unsweetened coffee / tea
- ordinary hydration
- routine rest that is already appropriate
- normal daily necessities

These create fake scarcity and make the reward system feel wrong.

## Good reward classes

Use rewards for controlled exceptions:

1. **Protected rest** — e.g. guilt-free leisure time.
2. **Constraint relief** — e.g. postpone a low-priority chore, never safety/bills/health commitments.
3. **Discretionary spending permission** — e.g. allow buying a non-essential item within a real-money cap.
4. **Goal-conflicting indulgence** — e.g. high-calorie meal with guardrails.
5. **Scarce free-time blocks** — e.g. half-day free time from productivity expectations.

## Pricing heuristic

Task coins are payment for effort/resistance:

- 15 min easy ≈ 5 coins
- 30 min medium ≈ 15 coins
- 45 min hard ≈ 25 coins
- 60 min very hard ≈ 35 coins
- optional/stretch task ≈ 1.5–2× equivalent daily task

Reward coins price the scarcity/risk of a permission, not the market price of the item:

- Low-risk rest permission: ~20 coins
- Chore deferral / mild constraint relief: ~35 coins
- Small discretionary purchase permission: ~60 coins
- Goal-conflicting food permission: ~90+ coins and explicit guardrails
- Half-day free-time block: ~150 coins
- Larger discretionary purchase permission: ~250+ coins

## Guardrails

For each reward, write descriptions in permission language:

- “允許自己花真錢買…” not “購買…”
- “coins 買的是許可權，不是代付” when real money is involved
- Include limits: NT$ cap, time cap, no binge, no late-night add-on, return to plan next day.

When reviewing pricing, evaluate:

1. Whether it conflicts with active goals.
2. Whether it creates an addictive/overuse loop.
3. Whether it helps recovery enough to justify the permission.
4. Whether it should be rare.
